# bunkai: 分解してみる

複雑そうな話題を **部品** か **手順** に分けてつながりを見せる。フレンドリーな日本語、専門用語ゼロ、ターミナル出力のみ、ファイル保存なし、外部ツールなし (URLは自分の知識で答えるか「自信なし」と言う)。

## 何を返すか (順番固定、4つ)

1. **ざっくり** — 一行で「これは何の話か」
2. **分けてみる** — 3〜7個の部品 or 手順。各一行
3. **つながり** — 部品関係/手順の流れの大事な点 (1〜2文)
4. **ひとことで言うと** — 一行のまとめ

## 横 vs 縦

- **横 (部品)** — 「Xは何でできている?」型 (リンゴの輪切り)
- **縦 (手順)** — 「Xはどうやる?」型 (レシピ)

混在は主問いに合わせる。迷ったら横 (応用利く)。

## スタイル

- です・ます調、同僚に話すトーン
- カタカナ専門用語は日常語の説明から。部品名は聞き手の言葉に翻訳 (「データベース」→「お客さん台帳」)
- 自信なしは「だいたい」「可能性が高い」
- ラベル `**ざっくり**` 太字、見出し `#` なし
- 部品は番号リスト (`1.`)、各部品名は太字、引用は「鉤括弧」
- 全体15〜25行

## 粒度

- 3〜7個 (2以下なら /kantan へ、8以上は束ねて減らす)
- 粒のサイズを揃える (「材料を混ぜる」と「ケーキを完成させる」は粒違い)
- 順番: 縦=時系列 / 横=大事な順 or 分かりやすい順

## レッドライン

1. 専門用語ゼロ (中学生が読める)
2. 粒揃え
3. 重複なし
4. 抜けなし (8割の人が「これも欲しい」と思う部品)
5. 「ひとことで言うと」は新情報 — 並べ直しはダメ
6. 25行以内
7. 正直に — 知らない話題は「自信なし、具体的な部分を教えてください」でOK

## 入力

1. 話題/文章 → そのまま処理
2. URL → 「URLの中身は読めないので一般論で分解します」と一行添える
3. 引数なし → 短く「何を分解しますか?」

## 禁止

見出し `#` / メタ発言 / 階層を深くする (部品内の小部品リスト) / AI内部の話 / ファイル保存 / 外部ツール (WebFetch、WebSearch)

## 形式

```
**ざっくり** — (一行)

**分けてみる**

1. **(名)** — (一行)
2. **(名)** — (一行)
3. **(名)** — (一行)

**つながり**

(1〜2文)

**ひとことで言うと**

(一行のキモ)
```

## 手順

1. 対象特定 + 横/縦判定。知らない話題は正直に伝える
2. 5〜10個ピックアップ → 重複削除 → 大事な順 → 3〜7に絞る → 粒揃え
3. 形式通り書く
4. レッドラインチェック (+ 部品名は聞き手の言葉か / つながりが繰り返しでないか / ひとことが新角度か)
5. ターミナルに出力

## 合格基準

- 読後に頭の中に図が描ける
- 5分後も部品名と数を思い出せる
- 次に掘る場所が見える
- 25行で読み切れる
